┌─────────────────────────────────────────────┐
│  Effective Registry                          │
│  ───────────────────────────                 │
│  Life:  computed, never persisted            │

│                                             │
│  Rules:                                      │
│  - MERGED view: system + user               │

## 2. Frozen API Contract

### 2.1 Paper (canonical, both system + user)

```typescript
interface Paper {
  /** Unique identifier.
   *  System papers: e.g. "A4", "Voucher240x140"
   *  User papers:   "user_<timestamp>" */
  readonly id: string;

  /** Display label shown in UI.
   *  System: "A4", "凭证纸 240×140mm"
   *  User:   user-specified, max 100 chars */
  readonly label: string;

  /** Paper width in millimeters. Range: [50, 1000] */
  readonly widthMM: number;

  /** Paper height in millimeters. Range: [50, 1000] */
  readonly heightMM: number;

  /** Source discriminator.
   *  "system" = from paper-registry.js (immutable)
   *  "user"   = from user-papers.json (mutable) */
  readonly source: 'system' | 'user';

  /** Creation timestamp (user papers only). ISO 8601. */
  readonly createdAt?: string;

  /** Last update timestamp (user papers only). ISO 8601. */
  readonly updatedAt?: string;
}
```

### 2.3 UserPaperStore

```typescript
interface UserPaperStore {
  /** Load all user papers from disk.
   *  Called once at app startup.
   *  Returns empty array if file does not exist or is corrupt. */
  load(): Promise<Paper[]>;

  /** Persist a new user paper.
   *  Input must pass validatePaper() before calling.
   *  id is generated by caller (not by store).
   *  @throws if id already exists (duplicate id is a contract violation). */
  create(paper: Paper): Promise<void>;

  /** Update an existing user paper by id.
   *  Input must pass validatePaper() before calling.
   *  @throws if id does not exist. */
  update(id: string, paper: Partial<Pick<Paper, 'label' | 'widthMM' | 'heightMM'>>): Promise<void>;

  /** Delete a user paper by id.
   *  Silently succeeds if id does not exist (idempotent). */
  delete(id: string): Promise<void>;
}
```

### 2.4 PaperRegistryProvider

```typescript
interface PaperRegistryProvider {
  /** System papers (immutable, from paper-registry.js) */
  readonly systemPapers: ReadonlyArray<Paper>;

  /** User papers (mutable, from UserPaperStore) */
  readonly userPapers: ReadonlyArray<Paper>;

  /** Effective registry = systemPapers + userPapers merged.
   *  This is a computed view. Never persisted. */
  readonly effectiveRegistry: ReadonlyArray<Paper>;

  /** Reload user papers from disk and rebuild effective registry. */
  reload(): Promise<void>;
}
```

 *  @returns ValidationResult */
function validatePaper(
  paper: Partial<Paper>,
  existingIds: ReadonlySet<string>,

## 3. ID Strategy

### Option A: `user_<timestamp>` ← **RECOMMENDED**

```
Format:  user_<epochMs>
Example: user_1718123456789
```

**Rationale:**
- Human-readable: instantly identifiable as user paper
- Sortable by creation time without an extra field
- No collision risk (millisecond precision)
- Lightweight: no uuid library dependency
- Stable across rename (id never changes on edit)

**Risk and mitigation:**
- Collision on rapid create (same millisecond): UserPaperStore.create() rejects duplicate id — caller retries with +1ms
- Deleted id reuse: No reuse needed. If user deletes "user_1718123456789" and creates a new one, the new id is different

### Option B: `uuid` ❌

Rejected. No UUID advantage is needed here — there is only one writer (single user on single machine), no distributed system, no sync requirement.

### Decision: Option A — `user_<timestamp>`

```
Pattern: /^user_\d+$/
Example: user_1718123456789
```

## 4. Lifecycle Operations

### loadUserPapers()

| Property | Value |
|----------|-------|
| When | App startup, before effectiveRegistry is built |
| Input | File path: `app.getPath('userData')/paper-registry/user-papers.json` |
| Output | `Paper[]` (empty array if file missing or corrupt) |
| Validation | Filters invalid entries silently (logs warning). Malformed JSON → empty array |
| Failure | Never throws. Corrupt data → empty array + console.warn |

### createUserPaper()

| Property | Value |
|----------|-------|
| When | User clicks "保存" in Add Paper dialog |
| Input | `{ label, widthMM, heightMM }` (id generated by caller) |
| Validation | BEFORE persistence: validatePaper() — label, widthMM, heightMM, duplicate id |
| Persistence | AFTER validation: append to file (atomic: write temp, rename) |
| Failure | Validation fail → throw `PaperValidationError` |
| Idempotent | No — duplicate id throws |

### updateUserPaper()

| Property | Value |
|----------|-------|
| When | User edits an existing paper's label/dimensions |
| Input | `{ id, label?, widthMM?, heightMM? }` |
| Validation | BEFORE persistence: validatePaper() on changed fields |
| Persistence | AFTER validation: rewrite file |
| Failure | Id not found → throw `PaperNotFoundError` |
| Idempotent | Yes — same update applied twice has no extra effect |

### deleteUserPaper()

| Property | Value |
|----------|-------|
| When | User clicks "删除" on a paper |
| Input | `id: string` |
| Validation | None (deletion is always allowed) |
| Persistence | Remove from array, rewrite file |
| Failure | Never throws — idempotent |
| Idempotent | Yes — delete non-existent id succeeds silently |

### buildEffectiveRegistry()

| Property | Value |
|----------|-------|
| When | After loadUserPapers(), after any create/update/delete |
| Input | `systemPapers: Paper[], userPapers: Paper[]` |
| Merge | 1. Filter user papers: `validatePaper()` == valid 2. Deduplicate: skip user papers whose id conflicts with system 3. Sort: system (original order) + user (createdAt DESC) |
| Output | `Paper[]` with `source` field set |
| Failure | Never throws. Silently drops invalid user papers. |

## 5. Migration Guarantees

### System paper IDs — UNCHANGED

```
A4            → A4             ✅ (same)
A5            → A5             ✅
A3            → A3             ✅
Letter        → Letter         ✅
Voucher240x140 → Voucher240x140 ✅
```

**Impact on existing saved settings:**

```json
// Database:  print609/database/Settings.json
{ "paperSize": "Voucher240x140" }

// This value is a paper ID string.
// Since system paper IDs never change, this setting continues to work.
// No migration script needed.
```

### What breaks if system paper IDs change

| Scenario | Impact |
|----------|--------|
| User changes `Voucher240x140` → `voucher` in registry | All users with `paperSize: "Voucher240x140"` saved → fallback to A4 |
| User adds a system paper with same ID as a user paper | User paper is silently dropped from effectiveRegistry |

### Guarantee

> **System paper IDs are permanent.**
> Adding a new system paper = new unique id.
> Deleting a system paper = id is never reused.
> Renaming a system paper = id stays, only label changes.

### Extension Strategy

```
All future fields follow the same pattern:
  1. Add optional field to Paper interface (backward compatible)
  2. UI shows field only when registry version >= threshold
  3. buildEffectiveRegistry() passes through unknown fields
  4. Consumers check field via optional chaining: paper?.defaultOrientation

This ensures:
  - No migration needed when adding fields
  - Old user-papers.json works with new code (missing fields = undefined)
  - New user-papers.json works with old code (ignored fields)
```

## 7. File Specification

### `user-papers.json` — Format

```json
{
  "version": 1,
  "papers": [
    {
      "id": "user_1718123456789",
      "label": "自定义尺寸 180×250",
      "widthMM": 180,
      "heightMM": 250,
      "source": "user",
      "createdAt": "2026-06-12T19:40:00Z",
      "updatedAt": "2026-06-12T19:40:00Z"
    }
  ]
}
```

- `version`: schema version for future migration
- `papers`: array of user-defined Paper objects
- File encoding: UTF-8 without BOM
